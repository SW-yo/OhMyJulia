### 2. 型を決める理由
- 型推論を一度だけで終わらせて、その後は迷わせない。
- 関数に「型」を教える
	計算を速くしたいときは「この関数は整数を受け取るよ」と教えてあげる。
```
# 型を指定しない（汎用的）
function add(a, b)
    return a + b
end

# 型を指定する（::型名）
function add_fast(a::Int64, b::Int64)
    return a + b
end
```
- 型がコロコロ変わる」のを避ける（型安定性）
	一つの変数の中身が、計算の途中で「整数だったり浮動小数点だったり」と変わると、Juliaは「次はどの計算機を使えばいいんだ？」と迷って遅くなる。
```
# 悪い例（型が不安定）
function slow_sum()
    s = 0  # 最初は整数
    for i in 1:10
        s = s + 0.5  # 途中で浮動小数点に変わる！
    end
    return s
end
```
- 型がコロコロ変わるとせっかくの**JITコンパイル**も機能しない。
- ゆくゆくは**型安定性**（**Type Stability**）が重要になる。

### 3. 型宣言
- 宣言方法
	- `::`  : 型アノテーション（型を宣言するだけ、値は別）
	- `=` : 代入（値を入れる）
- 型アノテーション
```
# 1つの型
x::Float64

# 複数の型を受け付ける
x::Union{Float64, Int64}

# タプル型
x::Tuple{Float64, Float64}

# 配列型
x::Vector{Float64}          # 1次元
x::Matrix{Float64}          # 2次元
x::Array{Float64, 3}        # 3次元

# 辞書型
x::Dict{Int64, Float64}

# ベクターのベクター
x::Vector{Vector{Float64}}
```
- 変数作成
```
# Vectorは[]で作れる
x = Vector{Float64}()      # Vector{Float64}の空配列
x = []    # これでも作れるが型がAnyになる（非推奨）

# Matrixは[]では作れない
x = Matrix{Float64}(undef, 3, 3)   # 3×3の未初期化
x = zeros(Float64, 3, 3)           # 3×3のゼロ行列

# Arrayも同様
x = Array{Float64, 3}(undef, 3, 3, 3)  # 3×3×3の未初期化

# DictはDict()で作れる
x = Dict{String, Float64}()               # キーがString、値がFloat64の空の辞書
x = Dict("apple" => 150.0, "banana" => 120.0) # 初期値を持った辞書（型は自動推論）

# Tupleは()で作れる
x = ()                                   # 空のタプル（Tuple{}型）、タプルは元々Any
x = (1, 2.5, "apple")                    # 初期値を持ったタプル（型は自動推論）
x = Tuple{String, Float64}("apple", 150.0) # 型を指定して作成

```
- これは厳密には型が違う。
	- `Array{Float64}(undef, 2, 1)` → `Matrix{Float64}`（2次元）
	- `Vector{Float64}(undef, 2)` → `Vector{Float64}`（1次元）
- `Dict()`の場合、型を固定してバグを防ぎたい場合は、`Dict{Keyの型, Valueの型}()` と指定するのがJuliaのベストプラクティス
- `[]`と`()`の使い分け
	- String() で空文字列を作るのと同じ感覚
	- `Vector{ETL1normalNN}()` はコンストラクタの呼び出し（型を使って新しいオブジェクトを作る）なので `()`
```
ETL1normalNN[]          # 空配列のリテラル、型を[]で囲む
Vector{ETL1normalNN}()  # Vectorコンストラクタを()で呼び出す
[1, 2, 3]               # Int64の配列リテラル
```
